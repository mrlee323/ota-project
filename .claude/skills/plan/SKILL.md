과제나 기능 요구사항을 분석해서 구체적인 개발 플랜을 만들어줍니다.

## 입력 처리

사용자 입력: `$ARGUMENTS`

- 특정 파일명이나 경로를 주면 그 파일을 요구사항으로 분석
- 텍스트로 요구사항을 주면 그대로 분석
- 인자 없으면 현재 프로젝트의 README, docs 폴더, 과제 관련 파일을 자동 탐색

## 분석 순서

### 1단계: 요구사항 파악

다음 파일들을 우선 탐색하세요:
- `README.md`, `docs/`, `requirements.*`, `spec.*`
- 과제 관련 파일: `TASK.md`, `ASSIGNMENT.md`, `*.spec.md`
- 기존 코드가 있다면 현재 구조 파악

요구사항에서 추출할 것:
- **기능 목록**: 무엇을 만들어야 하는가
- **제약 조건**: 사용해야 할 기술, 금지 사항, 성능 요구
- **엣지 케이스**: 예외 처리가 필요한 상황

### 2단계: 설계 결정

구현 전에 먼저 정해야 할 것들:
- **타입/인터페이스 설계**: 데이터 구조를 먼저 정의
- **컴포넌트/모듈 경계**: 어디서 어디까지가 하나의 단위인가
- **의존성 방향**: 어떤 모듈이 어떤 모듈을 알아야 하는가
- **상태 관리**: 어디서 상태를 들고 있을 것인가

### 3단계: 작업 쪼개기

"한 번에 하나씩 테스트 가능한 크기"로 작업을 분해하세요.
너무 크면 테스트하기 어렵고, 너무 작으면 비효율적입니다.

좋은 단위의 기준:
- 30분~2시간 안에 구현 + 테스트 가능
- 독립적으로 동작 확인 가능
- 명확한 입력/출력이 있음

# [기능명] 개발 플랜

## 요구사항 요약
- 핵심 기능 1
- 핵심 기능 2
- 제약 조건 (있는 경우)

## 타입 / 인터페이스 설계
```typescript
// 구현 전에 이 타입들을 먼저 정의
interface Foo {
  id: string
  // ...
}
```
타입을 먼저 잡으면 구현 방향이 명확해지기 때문에 항상 먼저 작성합니다.

## 디렉토리 구조 (변경/추가 파일)
```
src/
├── 새로 만들 파일.ts     # 역할 설명
├── 수정할 파일.ts        # 수정 내용
└── ...
```

## 개발 순서 (체크리스트)

작은 단위부터 바깥쪽으로 쌓아 올리는 순서입니다.

- [ ] **1. 타입 정의** — `src/types/foo.ts`
  - `Foo` 인터페이스, `Bar` 타입 정의
- [ ] **2. 유틸 함수** — `src/utils/fooUtils.ts`
  - `formatFoo()`, `validateFoo()` 구현
- [ ] **3. 핵심 로직** — `src/hooks/useFoo.ts`
  - API 연동, 상태 관리
- [ ] **4. UI 컴포넌트 (작은 것부터)** — `src/components/FooItem.tsx`
  - 단순 표시 컴포넌트 먼저
- [ ] **5. UI 컴포넌트 (조합)** — `src/components/FooList.tsx`
  - 작은 컴포넌트 조합
- [ ] **6. 페이지 연결** — `src/pages/FooPage.tsx`
  - 전체 조립

## 작성 원칙

- **타입 먼저**: 타입 설계가 잘 되면 구현은 자연스럽게 따라옴
- **안쪽부터 바깥쪽으로**: 유틸 → 훅 → 컴포넌트 → 페이지 순서
- **의존성 없는 것부터**: 다른 모듈에 의존하지 않는 것을 먼저
- **과도한 설계 금지**: 지금 당장 필요한 것만 설계, 미래 확장성은 나중에
