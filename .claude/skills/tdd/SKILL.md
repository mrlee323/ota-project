테스트 주도 개발(TDD) 사이클을 단계별로 진행합니다.

## 입력 처리

사용자 입력: `$ARGUMENTS`

- 특정 함수/컴포넌트명을 주면 그것에 대한 TDD 진행
- 파일 경로를 주면 그 파일의 테스트 작성
- 인자 없으면 현재 `/plan` 체크리스트에서 다음 작업 항목을 확인하거나, 사용자에게 무엇을 만들지 질문

## TDD 사이클

```
Red   → 실패하는 테스트 먼저 작성
Green → 테스트를 통과하는 최소한의 코드 작성
Refactor → 동작은 유지하면서 코드를 개선
```

이 사이클을 **하나의 기능 단위**씩 반복합니다.
너무 많은 것을 한꺼번에 하려 하지 말고, 한 번에 하나의 테스트만 통과시키세요.

---

## 진행 방식

### Phase 1: 테스트 케이스 설계

구현 전에 "어떤 동작을 검증할 것인가"를 먼저 정리하세요.

좋은 테스트 케이스의 기준:
- **정상 케이스**: 일반적인 입력에서 기대한 결과가 나오는가
- **경계 케이스**: 빈 값, 최대/최소값, null/undefined
- **에러 케이스**: 잘못된 입력, 네트워크 실패, 예외 상황

테스트 목록을 먼저 주석으로 나열하세요:
```typescript
describe('formatPrice', () => {
  // ✅ 정상: 1000 → "1,000원"
  // ✅ 정상: 0 → "0원"
  // ⚠️ 경계: 음수 → 에러 or "0원"?
  // ⚠️ 경계: 소수점 → 버림? 반올림?
  // ❌ 에러: null/undefined → 에러 던지기
})
```

경계 케이스와 에러 케이스에서 어떻게 동작해야 할지 불명확하면, 구현 전에 사용자에게 확인하세요.

### Phase 2: Red — 실패하는 테스트 작성

아직 구현이 없는 상태에서 테스트를 먼저 작성합니다.
테스트가 실패하는 것을 확인한 후 구현으로 넘어갑니다.

```typescript
it('1000을 "1,000원"으로 포맷한다', () => {
  expect(formatPrice(1000)).toBe('1,000원')
})
```

테스트 실행해서 실패 확인:
```bash
npm test -- --run [테스트파일명]
```

### Phase 3: Green — 최소 구현

테스트를 통과하는 **가장 단순한** 코드를 작성합니다.
지금 당장 필요하지 않은 최적화나 확장성은 고려하지 마세요.
"이게 너무 단순한 거 아닌가?" 싶을 정도로 작성해도 됩니다 — 리팩터링은 다음 단계에서 합니다.

테스트 통과 확인 후 다음 테스트 케이스로 이동합니다.

### Phase 4: Refactor — 개선

테스트가 모두 통과하는 상태에서:
- 중복 제거
- 변수명/함수명 개선
- 로직 단순화
- 성능 개선 (필요한 경우)

리팩터링 후 반드시 테스트가 여전히 통과하는지 확인합니다.

## 출력 형식

각 단계별로 다음을 명확히 보여주세요:

### 테스트 케이스 목록 (시작 전)
```
[기능명] 테스트 케이스
━━━━━━━━━━━━━━━━━━━━
✅ 정상 케이스
  - [입력] → [기대 출력]
⚠️  경계 케이스
  - [입력] → [기대 동작]
❌ 에러 케이스
  - [입력] → [에러/기본값]
```

### 각 사이클 진행 중
```
🔴 RED: [테스트 코드]
      → 실패 확인: ✓

🟢 GREEN: [구현 코드]
         → 테스트 통과: ✓

🔵 REFACTOR: [변경 내용]
            → 테스트 여전히 통과: ✓
```

### 완료 후 요약
```
완료된 테스트: X/Y
다음 작업: [체크리스트의 다음 항목]
```

## 컴포넌트 테스트 (React/Vue)

UI 컴포넌트를 테스트할 때의 원칙:
- **무엇을 렌더링하는가**: 사용자가 보는 텍스트, 버튼 등
- **상호작용**: 클릭, 입력 등의 이벤트
- **상태 변화**: 인터랙션 후 UI가 어떻게 바뀌는가

구현 세부사항(내부 상태, 특정 함수 호출 여부)은 테스트하지 마세요.
사용자 관점에서 "이게 올바르게 동작하는가"를 테스트합니다.

```typescript
// ❌ 구현 세부사항 테스트 (피하기)
expect(component.state.isLoading).toBe(false)

// ✅ 사용자 관점 테스트 (권장)
expect(screen.queryByText('로딩 중...')).not.toBeInTheDocument()
```

## 테스트 환경 파악

시작 전에 프로젝트의 테스트 설정을 확인하세요:
- 테스트 프레임워크: `vitest` / `jest`
- 기존 테스트 파일 위치와 네이밍 패턴: `*.spec.ts` / `*.test.ts`
- 테스트 유틸: `@testing-library/react`, `msw` 등
- 기존 테스트 파일을 읽어 프로젝트의 테스트 스타일을 파악하고 동일하게 작성
